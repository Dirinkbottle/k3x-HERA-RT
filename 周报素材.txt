测试方法：
  使用 riscv64-unknown-linux-musl-gcc -march=rv64gcv 编译，运行时自动从 vlenb CSR
  读取硬件 VLEN，以 m8 寄存器组（v0-v7, v8-v15, v16-v23, v24-v31）为单位，
  4 条 vle8.v / vse8.v 覆盖全部 32 个向量寄存器。每寄存器填入唯一可识别 pattern。

测试点：

  Test-1（同线程 yield 恢复）
    主线程填满 32 个 vreg → sched_yield → 读回比对。
    验证：同一线程主动让出 CPU 后，OS 能正确保存并恢复其向量上下文。

  Test-A（新线程初始状态）
    worker 线程启动后直接读出 vreg 值，与全零比对。
    验证：新线程的向量寄存器被 OS 初始化为零，不会泄漏父线程数据。

  Test-C（跨线程 yield 恢复）
    worker 填入自己的 pattern → sched_yield → 读回比对。
    验证：多线程并发使用向量单元时，OS 不会混淆不同线程的向量状态。

  Test-3（跨线程隔离）
    worker 退出后，主线程再次读出自己的 vreg，与最初写入的 pattern 比对。
    验证：子线程对向量寄存器的修改不会污染父线程的上下文。

结果：4 项全部 PASS，starry 的向量上下文切换（保存/恢复/初始化/隔离）正确。

5. llama.cpp后端搭建bridge跑fp16的openai-gpt2-124M-F16.gguf

目标:
  先不做完整的ggml新后端,而是在llama.cpp现有CPU后端里加一个实验性fast path.
  只要遇到满足条件的MUL_MAT节点,就尝试把这个matmul提交到StarryOS的/dev/k3_airunner.
  如果任何条件不满足,或者/dev/k3_airunner不可用,就直接回退到llama.cpp原来的CPU路径.

为什么先选openai-gpt2-124M-F16.gguf:
  一开始考虑过tinyllama-15M-stories-Q2_K.gguf,但是Q2_K量化会涉及ggml量化块解包、scale/min等逻辑,
  这会把第一阶段问题扩大成"量化格式lowering + 后端接入 + 内核运行时调度".
  所以第一阶段改成F16 GPT-2,这样核心问题简化成:
    ggml的F16/F32输入 -> 转成F32共享tensor -> k3_airunner提交F32 matmul -> 输出回写ggml dst.

思路:
  1. 不直接把ggml malloc出来的tensor指针交给内核.
     因为当前StarryOS的k3_airunner内核实现只接受MAP_SHARED用户内存,
     map_user_to_kernel会检查用户VMA是否来自shared backend.
     llama.cpp自己的内存不是这个来源,所以直接传一定会失败.

  2. bridge内部重新分配k3_aiRuntime::Tensor.
     TensorManager::alloc_tensor()底层使用MAP_SHARED匿名mmap,
     这样内核可以把这块用户态共享内存映射成kernel alias.
     C侧只负责把ggml src0/src1拷贝成连续F32 buffer;
     Rust bridge再把这些F32数据拷贝到共享tensor中.

  3. 第一阶段只支持最小集合:
       src0 type: F16或F32
       src1 type: F16或F32
       dst  type: F32
       tensor必须是2D
       src0/src1第0维必须是dense row
     不满足就return 0,让ggml原CPU路径继续跑.

  4. ggml MUL_MAT的维度语义要非常小心.
     ggml_mul_mat(a, b)的结果维度是:
       dst.ne[0] = a.ne[1]
       dst.ne[1] = b.ne[1]
     CPU实现里实际等价于:
       dst[col_of_b, row_of_a] = dot(a[row_of_a], b[col_of_b])
     对K3 matmul attr来说可以整理成:
       lhs = src0, shape m x k
       rhs = src1, shape n x k, 作为rhs_transposed输入
       out = m x n
     这里踩了一个坑: ggml的dst第0维连续,所以K3输出stride必须是:
       out_row_stride = 1
       out_col_stride = dst->nb[1] / sizeof(float)
     不能按普通C row-major写成out_row_stride=n,out_col_stride=1,
     否则结果会按另一种二维布局写回.

  5. target_hint先使用PREFER_CPU.
     当前k3_kernel_backend里的F32 matmul只支持CPU fallback.
     A100/X100路径现在主要是int8->i32 IME matmul.
     所以F16 GPT-2这条链路现阶段的意义是:
       llama.cpp -> ggml hook -> k3_airunner ABI -> scheduler -> kernel backend
     先把整条软件通路打通,后面再把F32/FP16算子替换成A100可执行实现.
